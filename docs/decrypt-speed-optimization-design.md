# 数据库解密速度优化设计

## 背景

当前全量解密流程在 `internal/chatlog/wechat/service.go` 中按文件串行执行：

1. 列出所有 `.db` 文件
2. 逐个调用 `DecryptDBFile`
3. 每个文件独立解密后替换目标文件

该模式在数据库文件数量较多时，总耗时接近“单文件耗时之和”，无法利用多核 CPU。

## 现状瓶颈评估

### 1) 文件级串行处理

- 位置：`Service.DecryptDBFiles()`
- 现状：`for` 循环串行解密每个 db 文件
- 影响：多核环境利用率低，全量解密时间线性增长

### 2) 已解密文件的回退复制采用整文件读入内存

- 位置：`Service.DecryptDBFile()` 中 `ErrAlreadyDecrypted` 分支
- 现状：`os.ReadFile` + `os.WriteFile`
- 影响：大文件场景内存峰值高，复制过程额外开销大

## 优化目标

1. 全量解密总耗时显著下降（尤其是多文件场景）
2. 不改变解密正确性与现有输出结构
3. 保持出错文件可跳过，整体任务继续执行
4. 控制并发度，避免过高 IO 抢占

## 方案设计

### A. 文件级并发解密（核心）

- 在 `DecryptDBFiles()` 内引入 worker pool
- 并发度策略：`workers = min(max(2, NumCPU), 8)`
- 分发模式：主 goroutine 投递文件路径，worker 调用 `DecryptDBFile`
- 错误策略：保留当前语义，单文件失败仅记录日志，不中断整体

### B. 已解密回退改为流式复制

- 在 `ErrAlreadyDecrypted` 分支改为 `io.Copy` 流式复制
- 避免整文件读入内存
- 降低大文件复制时的内存与 GC 压力

## 兼容性与风险

### 兼容性

- 对外接口不变
- 输出目录结构不变
- 失败处理行为保持兼容（继续处理其他文件）

### 风险与缓解

1. **并发过高导致磁盘竞争**
   - 使用并发上限 `8`
2. **替换过程文件句柄冲突**
   - 继续沿用现有 `replaceDB` 重试机制
3. **单文件失败影响整体**
   - 维持“记录失败并继续”策略

## 验证计划

### 功能验证

1. 全量解密后可正常启动 HTTP 查询
2. 随机抽样多个 db 文件可被 SQLite 正常读取
3. 自动解密场景下新增/更新 db 可正常落盘

### 性能验证

对比优化前后在同一机器、同一数据目录：

1. 全量解密总耗时（秒）
2. 单位时间处理文件数
3. 峰值内存（重点观察大文件回退复制场景）

## 回滚策略

若发现兼容性问题，可快速回滚到该优化前提交；由于接口未改，回滚风险较低。
